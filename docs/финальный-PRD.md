# Финальный PRD: VetCRM

Документ описывает реализованный объём продукта VetCRM — REST API для учёта в ветеринарной клинике.

---

## 1. Цель продукта

VetCRM — бэкенд-система для учёта клиентов, питомцев, приёмов, медицинских записей, уведомлений и отчётов. Реализован в виде модульного монолита на .NET 8 с единой точкой входа (API) и разделением по доменным модулям.

---

## 2. Стек и архитектура

- **Платформа:** .NET 8, C#
- **БД:** PostgreSQL 18.1
- **ORM:** Entity Framework Core
- **Аутентификация:** JWT (access + refresh), BCrypt для хеширования паролей
- **Архитектура:** модульный монолит; модули общаются только через контракты (Application.Contracts)

Модули: **Clients**, **Pets**, **Identity**, **Appointments**, **MedicalRecords**, **Notifications**, **Reports**. Общий слой **SharedKernel** (доменные исключения, контракты).

---

## 3. Реализованные модули и функции

### 3.1. Identity (учётные записи и аутентификация)

- Регистрация пользователей с ролями: Admin, Veterinarian, Receptionist (самостоятельная регистрация только для Veterinarian и Receptionist).
- Создание пользователей администратором (в т.ч. Admin).
- JWT: выдача access- и refresh-токенов при логине; обновление access по refresh-токену.
- Хранение refresh-токенов в БД.
- Восстановление пароля: forgot password (отправка токена сброса), reset password по токену.
- CRUD пользователей (Admin): создание, просмотр списка/по id, обновление (имя, роль), отключение (disable).
- Роли: Admin, Veterinarian, Receptionist. Статусы пользователя: Active, Disabled.

### 3.2. Clients (клиенты)

- Создание клиента: ФИО, телефон (уникальный), email, адрес, заметки.
- Список клиентов с поиском (по имени, телефону, email), пагинацией и фильтром по статусу (Active, Archived).
- Получение клиента по id.
- Обновление данных клиента.
- Архивация клиента.

### 3.3. Pets (питомцы)

- Создание питомца: кличка, вид, дата рождения, привязка к клиенту (опционально).
- Получение питомца по id (эндпоинт-заглушка).

### 3.4. Appointments (приёмы)

- Создание приёма: питомец, клиент, ветврач (опционально), время начала/окончания, причина. Проверка пересечений по ветврачу.
- Список приёмов по дате с опциональным фильтром по ветврачу.
- Перенос приёма (reschedule): новые время начала и окончания.
- Отмена приёма (cancel).
- Завершение приёма (complete): создаётся медицинская запись с жалобой, диагнозом, планом лечения, назначениями, вложениями.

### 3.5. MedicalRecords (медицинские записи)

- Медицинские записи создаются при завершении приёма (из модуля Appointments).
- Список медицинских записей по питомцу: `GET /api/pets/{petId}/medical-records`.
- Получение медицинской записи по id.
- Обновление медицинской записи (жалоба, диагноз, план, назначения, вложения).
- Добавление вакцинации к записи: название вакцины, дата прививки, следующая дата, серия, производитель.

### 3.6. Notifications (уведомления)

- Фоновая логика: напоминания о предстоящих вакцинациях (NextDueDate в пределах заданного окна).
- Ручной запуск отправки напоминаний: `POST /api/notifications/send` — создаёт записи в журнале и вызывает заглушки отправки (Email/SMS).
- Журнал напоминаний: список по типу, статусу, периоду дат.
- Доступ к API уведомлений: только Admin.

### 3.7. Reports (отчёты)

- **Приёмы за период:** список приёмов за указанный интервал дат, пагинация, ограничение периода (не более 1 года).
- **Просроченные/скоро просроченные вакцинации:** вакцинации с NextDueDate в пределах заданного окна (до 7 дней), с контактами владельца; пагинация.
- Доступ к отчётам: только Admin.

---

## 4. Защита API и роли

- Эндпоинты аутентификации (register, login, refresh, forgot-password, reset-password) — без авторизации.
- Остальные эндпоинты требуют JWT (Bearer). Роли заданы атрибутами `[Authorize(Roles = "...")]`:
  - **Admin:** пользователи, отчёты, уведомления, полный доступ к клиентам, питомцам, приёмам, медзаписям.
  - **Receptionist:** создание/обновление/архивация клиентов, создание питомцев, создание и перенос приёмов.
  - **Veterinarian:** просмотр клиентов/питомцев/приёмов, завершение приёмов, просмотр/обновление медзаписей, добавление вакцинаций.

---

## 5. Инфраструктура и окружение

- **Docker Compose:** один сервис PostgreSQL 18.1, база `vetcrm`, порт 5432.
- **Миграции БД:** ручное выполнение SQL-скриптов из `docs/sql` (001–007 или 000_Full_Schema + остальные по нумерации).
- **Seed (только Development):** `POST /api/dev/seed` — создаёт Admin, ветврача, клиентов, питомцев, приёмы (в т.ч. завершённый с медзаписью). Учётные данные Admin: `admin@vetcrm.local` / `Admin123!`.

## 7. Исключения и обработка ошибок

Глобальный `ExceptionHandlingMiddleware` маппит доменные исключения в HTTP-коды и JSON (type, title, status, detail, traceId):

- ClientNotFoundException, PetNotFoundException, AppointmentNotFoundException, MedicalRecordNotFoundException, UserNotFoundException → 404
- DuplicatePhoneException, DuplicateEmailException, AppointmentConflictException → 409
- InvalidCredentialsException → 401
- AccountDisabledException → 403
- ReportPeriodTooLongException и прочие DomainException → 400
